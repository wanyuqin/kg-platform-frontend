# pipeline — 入库流水线

> **溯源**：技术设计文档 八、附录 A / B；设计文档 三、四
> **代码入口**：`app/pipeline/`（parser.py / validators.py / sensitive.py / content_hash.py）
> **关联 ADR**：ADR-0005、ADR-0006、ADR-0007、ADR-0016、ADR-0017
> **最后同步**：2026-07-04

## 总则

P1 流水线**全同步执行**（无消息队列，ADR-0017）：表单与上传在请求内完成 解析 → 校验 → 敏感检测 → 去重 → 发布，失败当场反馈给提交人——对应设计 3.1"不通过当场拒收"的交互要求。零 LLM（ADR-0006），全部环节为确定性规则。

## Markdown 解析与切分（parser.py，技术 8.1）

输入约束：UTF-8、≤2MB、CommonMark。切分规则：

- 一级标题 `#` 为条目边界，一文件多条按此切分（FAQ / 术语 / 案例典型多条，其余类型 1:1 也走同一逻辑）；无一级标题时整文件视为一条；
- 条目内二级标题 `##` 的文本为模板段名，与附录段名映射精确匹配（先 trim，全角 / 半角冒号与空格归一化）；未知段名报 blocking 错误；段下内容即该段取值；
- 一级标题文本默认作为 title；FAQ 以"标准问法"段覆盖。

每条解析结果落 import_item，validation 字段记录 `[{rule, level: blocking | warning, message}]`；blocking 条目在预览确认页不可勾选，提示附 `GET /api/templates/{type}.md` 模板下载链接。

## 模板完整性校验（validators.py，技术 8.2）

校验器按类型硬编码为纯函数（与模板一起冻结，ADR-0007；改动走发版），每条规则配正反单测：

| 类型 | 规则 | 级别 |
|-|-|-|
| 通用 | 必填段缺失或内容为空 | blocking |
| 通用 | 悬空指代：正文命中 `参见\|见原文\|如上所述\|上述文档\|详见附件` | blocking |
| faq | 相似问法不足 2 条（降级不阻塞） | warning |
| sop | 操作步骤非有序列表，或存在不含"预期"描述的步骤 | blocking |
| sop | 命中高危动作词（删除 / 退款 / 资金 / 扣款 / 对客生效）且未填回滚方式 | blocking |
| policy | 生效时间缺失（过期复审机制的输入） | blocking |
| product | 能力边界缺"支持"或"不支持"子项 | blocking |
| case | 触发条件与根因为"未知" | warning（P1 记入 risk_note；P2 转中风险） |
| term | 定义中出现术语名自身（循环引用，整词匹配） | warning |

## 敏感信息检测（sensitive.py，技术 8.3）

正则为主、不调用 LLM。P1 来源均为同步提交，命中即拒绝并返回片段位置，由提交人脱敏后重提；P2 飞书自动同步的异步来源才需要隔离区。清单以代码常量交付：

| 项 | 规则要点 |
|-|-|
| 手机号 | `(?<!\d)1[3-9]\d{9}(?!\d)` |
| 身份证号 | `(?<!\d)\d{17}[\dXx](?!\d)` |
| 云凭证 AK | `AKIA[0-9A-Z]{16}`、`AKLT[\w-]{16,}`、`LTAI[A-Za-z0-9]{12,}` |
| Secret 赋值 | `(secret\|password\|api[_-]?key)\s*[:=]\s*\S{8,}`（忽略大小写） |
| 内网 IP | `10\.\d+\.\d+\.\d+`、`192\.168\.\d+\.\d+`、`172\.(1[6-9]\|2\d\|3[01])\.\d+\.\d+` |

## content_hash 与去重（content_hash.py，技术 8.4）

规范化定义（hash 输入）：按模板段顺序重排 → 各段 trim → 连续空白折叠为单个空格 → 段间以两个换行连接 → SHA-256 取 hex。去重靠 `uq_knowledge_hash` 部分唯一索引兜底（非 archived 范围），冲突返回 409 及已存在的 kid，前端提示与哪条重复（设计 4.3.3 三级漏斗的第一级）。

## 发布事务（技术 8.4）

1. PG 事务：取号生成 kid（新建时，规则见 [domain.md](domain.md)）→ 写 / 更新 knowledge（status=published，index_state=indexing，version=n）→ 插入 knowledge_version 快照 → COMMIT；
2. 调用 OpenViking `content/write`（同 URI 幂等覆盖）；写入成功后由 scheduler 轮询索引就绪，置 index_state=ready；
3. 写入失败：index_state=failed，scheduler 指数退避重试（初始 1min，最多 10 次），仍失败告警平台管理员。期间该知识 **read 可用**（内容在 PG 快照）、**search 不可见**，控制台按 index_state 展示"索引中 / 失败重试"。

一致性原则：PG 是唯一事实来源（ADR-0002），search 结果一律回查 PG 过滤，OpenViking 中可能残留的孤儿文件不影响正确性，由 P3 对账任务清理。

## 附录：六类模板的段名映射（技术 附录 A）

解析器段名匹配基准，与设计文档附录《知识类型模板全集》一一对应（✅ 必填；标注"可填×"的段必须出现、允许填占位值）。顺序不做强约束、入库时按下表顺序重排：

| 类型 | 段名（重排顺序） |
|-|-|
| faq | 标准问法✅ ・ 相似问法✅（≥2 条）・ 标准答案✅ ・ 适用条件✅（可填"无限制"）・ 例外情况 |
| sop | 目标与适用场景✅ ・ 前置条件✅（可填"无"）・ 操作步骤✅ ・ 异常与分支处理✅（可填"无"）・ 完成标志✅ ・ 回滚方式（高危动作时必填）・ 注意事项 |
| policy | 一句话摘要✅ ・ 适用范围✅ ・ 规则条款✅ ・ 例外条款✅（可填"无"）・ 生效 / 失效时间✅ ・ 罚则与违规处理 ・ 制度依据来源 |
| product | 功能定义✅ ・ 适用版本 / 套餐✅ ・ 能力边界✅（须含"支持"与"不支持"）・ 使用入口 ・ 限制与配额 ・ 常见误解澄清 |
| case | 问题现象✅ ・ 触发条件与根因✅（未明确须填"未知"）・ 排查步骤✅ ・ 解决方案✅ ・ 影响范围 ・ 预防措施 |
| term | 术语名✅ ・ 定义✅ ・ 同义词 / 别名✅（可填"无"）・ 使用示例 ・ 易混淆术语辨析 |

标准模板示例（FAQ 等）见技术文档附录 B，交付物为 `GET /api/templates/{type}.md`。
