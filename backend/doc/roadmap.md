## MVP 定义

**一条知识从录入到被 Agent 检索引用的最短闭环。**交付给 Agent 侧的东西只有三样：两个检索接口（含 API Key）、一个已灌入知识的试点 domain（免单域）、一页接入说明。

```text
owner 表单录入 / 上传规范 Markdown（规则解析，零 LLM）
  → 完整性校验 + 敏感信息正则 + content_hash 去重 → 低风险自动发布
  → content/write 写入 OpenViking（domain/type 目录）
  → Agent：POST /v1/search → GET /v1/knowledge/{kid}
    （API Key + domain 白名单 + 审计留存，第一天即开启）
```

## 三阶段里程碑

原图为设计第九章里程碑画板，范围表如下：

| 阶段 | 范围 | 明确不含 | 验收标准 |
|-|-|-|-|
| **Phase 1 · 最小 MVP** | 表单 + Markdown 两通道（零 LLM）；完整性校验 + 敏感正则 + hash 去重；低风险自动发布；OpenViking 入库；search / get + API Key；审计留存首日开启；控制台最小集（domain 注册 + 知识管理） | 飞书同步、LLM、审核卡片、向量 / LLM 冲突检测、MCP、打标界面 | 免单域客服 Agent 完成"录入 → 检索 → 引用回答"闭环；低风险自动发布占比 >80% |
| **Phase 2 · 飞书接入与审核完整化** | 飞书单文档注册 + 事件/轮询双通道同步；LLM 重组兜底；风险矩阵 + 飞书卡片审核；重拆对齐；MCP Server；查询记录打标 | feedback API（ADR-0010） | 运营"只改飞书原文"跑通：原文更新 → 自动重拆 → 低风险自动生效，全程无人工介入 |
| **Phase 3 · 治理自动化** | 过期复审、月度冲突扫描、报表全集（含未审阅积压监控）、类型级 top_k 精调 | — | 治理任务自动运转一个月：零命中下架、过期排除、冲突扫描各完成一轮，LLM 用量在硬上限内 |

两处相对早期交接文档的调整：① 审计留存提前至 Phase 1 首日——它是治理数据主源，晚开一天少一天数据；② 冲突检测的向量预筛与 LLM 灰区判定随 Phase 2 一起上，保持 Phase 1 严格零 LLM。

## 质量线与验收口径（技术 十三）

- **检索质量**：免单域业务方提供 50～100 组 golden set（query → 期望命中 kid）；P1 放行线 **Recall@5 ≥ 0.85、Top1 命中率 ≥ 0.60**。未达线先调语料质量与 top_k 配置再复评，不引入平台侧重排开发。评测脚本进仓库作回归基线。
- **性能目标**（50 并发、万级知识库）：search P95 ≤ 800ms（含 OpenViking 检索）；read P95 ≤ 200ms（走 PG 快照）；限流精确 429 不放大到上游。
- **发布占比口径**：低风险自动发布占比 = 自动发布成功数 ÷ 提交入库总数；同步校验被当场拒绝的不计入分母（拒收是入口质量控制，不是审核拦截）。统计窗口为试点启动后 2 周，目标 >80%。

## PoC 验证清单（2026-07-04 完成，结论已回填 [storage.md](modules/storage.md)）

- [x] content/write 覆盖写语义与幂等性——upsert 组合（replace→404→create）
- [x] 写入 → 可被检索的实际延迟——约 16s，符合"秒～分钟级"假设
- [x] 多目录前缀检索的接口形态——target_uri 数组，单次多前缀可行
- [x] L0/L1 生成状态的查询方式——无按条接口，就绪判据 = find probe level=2 命中（设计预判的兜底分支）
- [x] 中文 L0 摘要质量抽检（20 条六类 mock 全部生成；发现 L0 语言不稳定问题——部分中文内容出英文摘要，线上网关复测，Gateway 可用 PG title 兜底）
- [x] 单目录批量检索延迟（avg 233ms / p95 378ms，800ms 预算内；500 条压测后置至真实数据灌入后）
- [ ] AGPLv3 开源合规评审发起（流程动作，用户线下跟进）

附加发现五项（用户级 key、维度冻结、level=2 过滤、账号随 workspace 丢失、L0 语言不稳定）详见 storage.md。
