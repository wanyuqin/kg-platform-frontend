# ADR-0017：P1 不引入 RocketMQ，后台任务用 APScheduler

- **状态**：已采纳
- **日期**：2026-07-04（MVP 技术设计阶段变更）
- **来源**：技术设计文档 1.2 变更③（设计文档第八章选型结论不变）

## 背景

设计文档第八章技术选型定了 RocketMQ 用于流水线投递 / 飞书事件 / 卡片超时（此前曾考虑 Celery，已移除；Redis 只做限流 / 缓存，不做队列）。但 P1 范围内没有异步业务消息——飞书同步与审核卡片均为 P2。

## 决策

P1 不部署 RocketMQ；流水线全同步执行，后台任务（重试、分区维护）用 APScheduler 单进程承载；RocketMQ 随 P2 接入。

## 理由

为不存在的消息流提前部署中间件是纯成本；审计写入等"若需强保证"的场景已预留切换点（写入口换 MQ、表结构不变），P2 接入无返工。这是实现层决策，不改变设计文档的选型结论。
