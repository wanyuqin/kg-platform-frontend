# 知识文件（source_doc）设计：条目归属与文件级 CRUD

> 日期：2026-07-04
> 状态：待评审
> 溯源：飞书设计文档 3.1（Markdown 上传）、4.1.4（重拆对齐）、7.2（拆分预览确认页）；本次浏览器实测发现 + 用户决策
> 关联决策：文件=管理容器（条目仍是生命周期原子）；统一源文档模型；所有条目必须属于文件；来源枚举英文

## 1. 背景与问题

当前 Markdown 导入将文件拆成条目后，文件本身即被丢弃：`import_batch` 只是一次性解析记录，条目虽有 `source_ref = "import:{batch}:{seq}"` 可溯源，但用户无法：

- 从"一份 FAQ 文档"的视角查看它拆出的全部条目；
- 对文件整体做生命周期管理（更新、续期、下架）；
- 在平台内粘贴文本创建知识（唯一入口是上传 .md 文件）；
- 在平台内直接编辑文件内容（改错字也要回本地改文件重传）。

## 2. 概念模型

```
domain（知识域）
  └── source_doc（知识文件，1..n）
        └── knowledge（知识条目，1..n）
```

- **知识文件**是管理容器：命名、归属 domain、单一知识类型、来源、整体操作入口。
- **知识条目**仍是平台的原子：状态机、版本、过期日、审核、检索命中全部保持条目级，已拍板决策不变。文件级操作 = 批量快捷方式。
- **所有条目必须属于一个文件**，不存在游离条目。表单创建改为"往文件里添加条目"。
- 文件的"内容"没有独立正文存储——**全文 = 文件下条目按 `doc_seq` 顺序拼合的 Markdown**（每条 `#` 标题 + `##` 模板段）。条目是唯一事实源，不存在正文与条目两份数据的一致性问题。
- kid、URI 规则不变（URI 仍为 domain/type 两级平铺）；Gateway 检索侧零改动。

## 3. 数据模型

### 3.1 新表 `source_doc`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGSERIAL PK | |
| name | VARCHAR(256) | domain 内唯一（UniqueConstraint(domain_code, name)） |
| domain_code | VARCHAR(32) FK → domain | |
| type | VARCHAR(16) | 六类之一，一个文件单一类型 |
| source | VARCHAR(16) | CHECK：`manual`（平台自建：粘贴/在线编辑/表单起建）/ `upload`（上传 .md）/ `feishu`（飞书文档，后续接入复用本表） |
| status | VARCHAR(16) | CHECK：`active` / `archived`，默认 active |
| created_by | VARCHAR(64) | |
| created_at / updated_at | TIMESTAMPTZ | |

### 3.2 已有表改动

- `import_batch` + `source_doc_id` BIGINT FK，**可空**：
  - 首次导入：预览阶段为空，"确认入库"时创建 source_doc 并回填（预览放弃不留悬空文件）；
  - 更新导入：批次创建时即指向已有文件。
  - 批次从此定位为"某文件的一次解析/更新记录"，文件详情页的变更历史直接查它。
- `import_item` + `align_action` VARCHAR(16)（`new` / `changed` / `unchanged` / `disappeared`，首次导入全为 new）、+ `match_kid` VARCHAR(64)（对齐命中的旧条目 kid）。消失条目也落 item 行（content 为空串、seq 排在解析条目之后），预览页有行可展示。`content` 列语义放宽为允许空串。
- `knowledge` + `source_doc_id` BIGINT FK，**非空**（存量回填后加约束）；+ `doc_seq` INT 非空（条目在文件内的顺序，用于拼合全文；每次确认对齐后按新文本顺序重写，表单追加取 max+1）。`source_ref` 保留，继续记录"来自哪个批次第几条"（表单添加记 `form:{user_id}`，不变）。

## 4. 入口与页面交互（CRUD 全景）

### 4.1 Create——三个入口

1. **粘贴文本**（主路径）：导入页第一步改双 tab：`粘贴文本`（textarea + 必填文件名）/ `上传 .md`（文件名默认取上传文件名，可改）。domain、类型选择照旧；解析、拆分预览、确认入库完全复用现有链路。粘贴 → source=`manual`；上传 → source=`upload`。
2. **上传 .md**：同上，仅入口不同。
3. **表单起建**：表单页"所属知识文件"必选——下拉列出该 domain 同类型 active 文件，或就地输入名称新建（source=`manual`）。新建文件与首条条目同事务落库；保存草稿同样要求选定/新建文件（草稿条目也须有归属），草稿 30 天清理只删条目不删文件，允许出现空文件。

### 4.2 Read——文件列表与详情

- **侧边栏新增「知识文件」**（位于知识管理之下）。
- **列表页**（按当前 domain 过滤）：名称 / 类型 / 来源 / 条目数（在架/总）/ 状态 / 最近更新 / 操作（查看 · 更新 · 整体续期 · 整体下架）。支持按类型 tab、状态筛选、名称搜索。
- **详情页**三区：
  - 基本信息卡：名称（可改）、domain、类型、来源、状态、创建人/时间、最近更新；
  - **条目视图 / 全文视图** 切换：
    - 条目视图：文件下条目列表（按 doc_seq），每行显示标题、状态、版本、过期日，点击进知识详情；行内保留条目级操作（续期/下架）；
    - 全文视图：条目拼合的 Markdown 只读渲染，所见即"这份知识文件现在的完整内容"，可一键复制；
  - 变更历史：历次批次（时间、操作人、来源方式、新增/变更/下架条数），点击可查该批次的解析明细。

### 4.3 Update——三种粒度

1. **在线编辑全文**（平台内闭环，主推）：详情页"编辑全文"→ 全文视图变为 Markdown 编辑器（预填当前拼合全文）→ 提交 → 后端按"粘贴新版本"处理：重拆 → 对齐 → 预览确认。用户改错字、补条目、删条目全部不出平台。
2. **粘贴新版本 / 重新上传**：详情页入口，适合"源头在外部文档"的用户把新版全文贴进来。与 1 走完全相同的对齐链路。
3. **条目级编辑**：知识详情页现有编辑能力不变（发新版本）。条目改动不回写文件——下次全文视图拼合时自然反映（条目是唯一事实源）。

### 4.4 Delete——下架语义，不做物理删除

- **文件整体下架**：确认弹窗（列出将下架的在架条目数）→ 批量下架全部在架条目 + 文件置 `archived`。归档文件在列表可筛选查看，详情只读。P1 不做解档恢复。
- **条目从文件"删除"**：即条目下架（仍挂在文件下，全文视图不再包含下架条目）。P1 不做条目跨文件移动/重新归属。
- 在线编辑/粘贴新版时删掉某条 → 对齐标记 disappeared → 默认下架（见 §5）。

## 5. 更新对齐规则（4.1.4 落地，P1 零 LLM）

新文本重拆后，按**标题精确匹配**（FAQ 用标准问法；标题先 trim 归一）对齐文件下现有**非下架**条目：

| 情形 | align_action | 预览默认勾选 | 确认后动作 |
|---|---|---|---|
| 标题匹配且 content_hash 相同 | unchanged | 不勾 | 无 |
| 标题匹配、内容变化 | changed | 勾 | 该条目发布新版本（version+1，走既有发布事务） |
| 新标题 | new | 勾 | 新条目入库（doc_seq 按新文本顺序） |
| 旧条目标题未出现在新文本 | disappeared | 勾（下架） | 下架 |

边界处理：

- **表单添加的条目**（source_ref 以 `form:` 开头）不在粘贴文本中属正常情况：标记 disappeared 但默认**不勾选**，行内注明"表单添加，未在新文本中"，避免每次更新被误下架。（在线编辑全文场景不受此影响——编辑器预填的全文包含它们。）
- **改标题**会被识别为"消失+新增"（新条目获得新 kid，旧条目下架）：P1 标题匹配的已知边界，预览页可见可人工纠正勾选；语义级对齐留给后续 LLM 兜底阶段。
- 对齐确认后按新文本顺序重写存活条目的 doc_seq；未在新文本中且保留在架的条目（如表单条目）排在末尾，保持原相对顺序。
- 更新预览底部汇总："预计：新增 X 条、更新 Y 条、下架 Z 条、不变 W 条"。
- 更新场景同样受审核策略约束：changed/new 条目按既有风险判定走绿色通道或进审核，与单条发布一致。

## 6. API

复用/扩展：

- `POST /api/knowledge/imports`：新增 `text` 字段（与 multipart file 二选一）+ `doc_name`（粘贴必填；上传默认文件名）。响应不变。
- `POST /api/imports/{batch_id}/confirm`：首次导入时创建 source_doc；回填条目 source_doc_id、doc_seq。同名校验两道：imports 创建时即预查 doc_name 给出即时报错，confirm 落库时依赖唯一约束兜底并发（409）。

新增（权限沿用 domain 三级角色，越权 404，与现有口径一致）：

| 接口 | 说明 |
|---|---|
| `GET /api/source-docs?domain=&type=&status=&q=` | 列表，含条目计数聚合 |
| `GET /api/source-docs/{id}` | 详情：基本信息 + 条目列表（按 doc_seq）+ 批次历史 |
| `GET /api/source-docs/{id}/content` | 拼合全文 Markdown（全文视图/编辑器预填共用） |
| `POST /api/source-docs/{id}/update` | body：text 或 file。重拆+对齐 → 生成带 align_action 的新批次，返回预览（确认仍走 `/api/imports/{batch_id}/confirm`） |
| `POST /api/source-docs/{id}/renew` | 整体续期：全部非下架条目过期日顺延（入参天数，默认 domain 的 default_ttl_days） |
| `POST /api/source-docs/{id}/offline` | 整体下架 + archived |
| `PATCH /api/source-docs/{id}` | 重命名（domain 内查重） |

跨文件内容重复仍按 content_hash 全局 409（重复知识不因分属不同文件而放行）。

## 7. 存量迁移（alembic，一次性）

1. 建 `source_doc` 表；三张表加列（约束先不加非空）。
2. 已确认的 import_batch 各生成一个 source_doc（name=file_name，source=`upload`，同名追加序号去重），回挂批次及其 result_kid 条目（doc_seq 取 item.seq）。
3. 其余条目（source_type=`manual`）归入每 domain 自动创建的「手工录入」文件（source=`manual`）。
4. 回填完成后 `knowledge.source_doc_id`、`doc_seq` 加非空约束。

当前库存量 4 条（1 表单 + 3 导入），迁移风险极小；迁移脚本仍按通用逻辑写，不依赖当前数据形态。

## 8. 测试口径

- 迁移：存量条目全部有归属、doc_seq 正确、约束生效。
- Create：粘贴/上传/表单起建三入口各一链路；粘贴缺文件名 400；同名文件 409。
- Read：列表聚合计数；详情条目排序；全文拼合与条目一致（含跳过下架条目）。
- Update：在线编辑与粘贴新版本走同一链路；对齐四类动作各至少一例；表单条目不默认下架；改标题=消失+新增；doc_seq 重写；changed 条目版本 +1 且快照正确。
- Delete：整体下架后条目状态与文件 archived；归档文件详情只读。
- 权限矩阵：三级角色 × 文件级操作；越权 404。
- 前端浏览器验收：双 tab 导入、文件列表/详情/全文视图、在线编辑 → 对齐预览四类徽标、整体操作确认弹窗。

## 9. 明确不做（P1）

- 条目跨文件移动 / 重新归属；
- 归档文件解档恢复；
- 文件级独立过期日（文件不持有生命周期字段，仅条目有）；
- LLM 语义对齐（标题改名识别）；
- 飞书文档接入（模型已预留 source=`feishu`，接入时复用本表与对齐链路）。
